1. Introduction 
1.1 Purpose 
This Software Requirements Specification (SRS) defines the complete functional, non-
functional, and system requirements for InvestorRadar, an AI-powered market 
intelligence platform built for ET AI Hackathon 2026, Problem Statement 6: AI for the 
Indian Investor. 
India has over 14 crore demat account holders. The vast majority make decisions based 
on tips and gut feel rather than data. InvestorRadar provides institutional-grade chart 
pattern intelligence and corporate event signal detection in plain-English, source-cited 
format that any retail investor can act on. 
1.2 Scope 
InvestorRadar comprises two core modules: 
•Chart Pattern Intelligence — real-time detection of 20+ technical patterns across all Nifty 
500 stocks, with backtested historical win rates and AI-generated explanations. 
•Opportunity Radar — continuous monitoring of BSE bulk/block deals, SEBI insider trade 
disclosures, and promoter activity to surface non-obvious pre-move signals. 
Out of scope for this submission: trade execution, brokerage integrations, regulated financial 
advice, and user portfolio tracking.

6. AI and LLM Layer 
6.1 Model Selection 
The system uses the claude-sonnet-4-6 model via the Anthropic Messages API for all 
LLM inference. This model is selected for its strong instruction-following, factual 
grounding, and cost-efficient performance at hackathon scale. Model string: claude-
sonnet-4-6. 
6.2 Prompt Architecture 
Each explanation request uses a two-part prompt structure: 
•System prompt: establishes persona (data-driven Indian market analyst), output format 
constraints (three length variants), citation requirements, disclaimer obligation, and 
grounding rules (no hallucinated statistics). 
•User prompt: injects all quantitative signal data as a structured JSON block, including all 
fields from the signals table and associated corporate events.

6.4 Output Quality Rules 
•All numerical claims must come from the injected context data — the LLM must not 
hallucinate statistics. 
•Every output must cite at least one source using the format: [Source: <source_name>, 
<date>]. 
•The one-liner must include the pattern common name, win rate at 15 days, and time 
horizon in plain English. 
•The mandatory disclaimer must appear as the final sentence of every paragraph and 
deep-dive output. 
•LLM outputs must not use financial jargon without immediately defining it for a general 
audience. 
6.5 Caching Strategy 
All three explanation lengths are generated in a single API call and stored in the signals 
table immediately. Redis is used as an L1 cache keyed on signal_id with a 24-hour TTL 
for one-liner and paragraph outputs. Deep-dive explanations are cached in Redis with a 
6-hour TTL. Cache misses on /explain/:id trigger an asynchronous LLM call; the frontend 
receives a 202 Accepted response and listens via SSE for completion.

8. System Constraints and Limitations 
8.1 Data Constraints 
•yfinance data is delayed by approximately 15 minutes. The system does not provide real-
time tick data and must label all price data with its delay. 
•BSE bulk deal data is only available after market close (~18:00 IST). Opportunity Radar 
signals based on this data are generated after hours. 
•SEBI EDGAR filings may have processing delays of up to 24 hours after the actual 
corporate event. 
•TA-Lib pattern functions require a minimum of 1–5 candles of history depending on the 
pattern. Stocks with insufficient history (recent IPOs, illiquid stocks) will yield fewer 
detected patterns. 
•Backtest sample counts will be low for rare patterns on illiquid stocks. These are flagged 
low_confidence=TRUE as per FR-BT-04. 
8.2 Regulatory Constraints 
•InvestorRadar does not constitute regulated financial advice under SEBI (Investment 
Advisers) Regulations, 2013. 
•All AI-generated content must include the disclaimer: 'This is not financial advice. Past 
patterns do not guarantee future results. Please consult a registered financial advisor 
before making investment decisions.' 
•The platform must not use non-public or material non-public information (MNPI). All data 
sources must be publicly accessible without authentication or privileged access. 
8.3 Cost Constraints 
•LLM API calls are limited to explanation generation only — no real-time inference on tick 
data. 
•Explanation caching in Redis and TimescaleDB must be implemented before any LLM 
feature is deployed. 
•The system must be deployable within the free or low-cost tier of Railway, Render, or 
equivalent PaaS for the duration of the hackathon.
